# 実装規約と上書き原則

Stellaris の mod 実装では、正しい構文より先に「どこへ置くか」「どう差し替えるか」を間違えやすい。ここでは、バニラの構成と既存 reference 群から導ける実装規約をまとめる。

## 基本ルール

- まず対象システムの専用 reference を読む。
- 同じ情報が複数箇所にあるときは、具体的なドメイン reference を優先する。
- バニラに似た既存パターンがあるなら、ゼロから設計せずそれに寄せる。

## 読み込み順と上書き

- 多くの `common/` / `events/` ファイルは ASCII 順に読まれる前提で考える。
- 追加型の mod は `zz_` などで後勝ちに寄せると扱いやすい。
- バニラの定義を直接潰すより、新規 ID 追加と参照差し替えを優先する。
- イベント ID は重複させない。重複時は「先に読まれた方だけ有効」という前提で設計する。
- バニラを差し替える必要があるときだけ、明示的な override 戦略を採る。

## 分岐の作り方

バニラでは、似た定義を丸ごと複製するより、差分だけを切り替える仕組みが多い。

- 政体や帝国タイプで建造物を切り替えるなら `convert_to` を検討する。
- 技術の見た目や効果差分なら `technology_swap` を検討する。
- job 名称やアイコンの差分なら `swappable_data` を検討する。
- 条件差分だけなら `triggered_*_modifier` や `triggered_desc` を検討する。

## パフォーマンス寄りの規約

- 頻繁に走るイベントは `is_triggered_only = yes` を優先する。
- `planet` / `pop` / `leader` / `system` / `starbase` 系イベントは `pre_triggers` を積極的に使う。
- `random_events` 側で重み変動を書きたくなったら、イベント本体の `weight_multiplier` を確認する。
- 重い共通処理は script 化して再利用し、必要なら `optimize_memory` を検討する。

## デバッグの考え方

- 変な UI 挙動は scope ミスか parse ミスを疑う。
- 空のイベントウィンドウ系は括弧対応や event 定義の破損を最優先で疑う。
- scope が読めないときは、汎用の debug 用 scripted effect / trigger パターンが有効。

## 迷ったときの順番

1. 対象ドメインの reference を読む。
2. そのドメインで分岐や再利用が必要か判断する。
3. 必要なら [18.scripted_effects・scripted_triggers・inline_scripts](18.scripted_effects・scripted_triggers・inline_scripts.md) を読む。
4. jobs / outputs / modifiers が絡むなら [19.経済カテゴリ・triggered_modifier・zones](19.経済カテゴリ・triggered_modifier・zones.md) を読む。
