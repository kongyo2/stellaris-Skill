# POP と職業のMod作成ガイド

このドキュメントでは、Stellaris における POP カテゴリ（社会階層）と職業のMod作成方法について解説します。

---

## ファイル構造

| 種類 | パス |
|------|------|
| POP カテゴリ | `common/pop_categories/*.txt` |
| 職業 | `common/pop_jobs/*.txt` |

## POP カテゴリ（社会階層）

POP カテゴリは社会における POP の位置を定義します。惑星人口ビューでは、職業は階層ごとにグループ化され、ファイル定義順でソートされます。

### バニラの階層構造

| 階層 | ランク | 用途 |
|------|--------|------|
| `ruler` | 4 | 支配者層 |
| `specialist` | 3 | 専門家層 |
| `complex_drone` | 3 | 複雑なドローン（集合意識） |
| `worker` | 2 | 労働者層 |
| `simple_drone` | 2 | 単純ドローン（集合意識） |
| `slave` | 2 | 奴隷 |
| `maintenance_drone` | 1 | メンテナンスドローン |
| `civilian` | 1 | 市民（失業中） |
| `purge` | 0 | 粛清対象 |

### データ構造

```pdx
<pop_category_key> = {
    rank = <int>                              # 社会的ランク (0-4)
    change_job_threshold = <float>            # 転職閾値 (1.15-2.0)
    keep_from_former_job = <yes/no>           # 失業後も階層を維持
    demotion_time = <int>                     # 降格に必要な日数
    display_category = <pop_category_key>     # UI表示用カテゴリ
    can_change_category = <yes/no/conditions> # カテゴリ変更可否
    
    # ウェイト計算（複数カテゴリ該当時の優先度）
    weight = {
        weight = <int>
    }
    
    # POP修飾子（住居・アメニティ使用量など）
    pop_group_modifier = {
        pop_housing_usage_base = <float>
        pop_amenities_usage_base = <float>           # 通常帝国
        pop_amenities_usage_no_happiness_base = <float>  # 集合意識
        pop_political_power = <float>
    }
    
    # 条件付きPOP修飾子
    triggered_pop_group_modifier = {
        potential = { <conditions> }
        <modifiers>
    }
    
    # 条件付き惑星修飾子
    triggered_planet_modifier = {
        potential = { <conditions> }
        <modifiers>
    }
    
    # 再定住設定
    allow_resettlement = { <conditions> }
    resettlement_cost = {
        trigger = { <conditions> }   # オプション
        <resource> = <int>
    }
    
    # 失業ペナルティ
    should_apply_unemployment_penalties = { <conditions> }
    unemployment_penalties = { <modifiers> }
    unemployment_resources = { <economy_unit> }
    
    # リソース（維持費・生産）
    resources = {
        category = <economy_category>
        upkeep = { <resources> }
        produces = { <resources> }
    }
    
    # カテゴリ割り当て条件（POPスコープ）
    assign_to_pop = { <conditions> }
}
```

### 詳細説明

#### ランクと転職

| プロパティ | 説明 |
|-----------|------|
| `rank` | 社会的ランク。高いランクの POP は低いランクの職業を取らない |
| `change_job_threshold` | 現在の職業よりこの倍率以上のウェイトを持つ職業がないと転職しない |
| `keep_from_former_job` | `yes` の場合、失業後も `demotion_time` 経過まで階層を維持 |
| `demotion_time` | 降格にかかる日数。専門家は 1800 日 |

#### 降格の仕組み

`keep_from_former_job = yes` の場合：
1. POP が失業すると、元の階層のまま維持
2. `demotion_time` が経過するまで、同等以上のランクの職業のみ受け入れ
3. 時間経過後、低ランクにリセット

複雑ドローンは `keep_from_former_job` を持たないため、即座にリセットされます。

#### 奴隷の仕組み

奴隷は特殊な階層で、カスタム `assign_to_pop` により高 `weight` を持ちます。POP の実際の階層は、職業の階層と `assign_to_pop` を持つ全階層から選択されます。

**例**: 債務奴隷の職人 POP
- 候補: 専門家、労働者、奴隷
- 奴隷が最高ウェイト → 消費財維持費は奴隷基準

### バニラ例: 専門家

```pdx
specialist = {
    rank = 3
    change_job_threshold = 1.25
    
    weight = {
        weight = 2
    }
    
    pop_group_modifier = {
        pop_housing_usage_base = 1
        pop_amenities_usage_base = 1
    }
    
    allow_resettlement = {
        always = yes
    }
    
    resettlement_cost = {
        energy = 250
    }
    
    resettlement_cost = {
        trigger = {
            owner = {
                NOT = { has_valid_civic = civic_corvee_system }
            }
        }
        unity = 25
    }
    
    triggered_planet_modifier = {
        potential = {
            exists = owner
            owner = { has_civic = civic_dystopian_society }
        }
        planet_crime_add = 1
    }
    
    inline_script = "pop_categories/social_classes_triggered_modifiers"
    
    resources = {
        category = pop_category_specialists
        inline_script = "pop_categories/living_standard_specialist_production"
    }
}
```

## 職業

職業は `common/pop_jobs/*.txt` で定義され、POP に割り当てられる仕事を指定します。

### データ構造

```pdx
<job_key> = {
    category = <pop_category_key>         # 所属階層
    is_capped_by_modifier = <yes/no>      # 修飾子で制限されるか（デフォルト: yes）
    first_come_first_served = <yes/no>    # 先着順割り当て
    can_set_priority = <yes/no>           # 優先度設定可能か
    can_be_automated = <yes/no>           # 自動化可能か
    contributes_to_diplo_weight = <yes/no> # 外交力に貢献するか
    ignores_sapience = <yes/no>           # 知性を無視するか
    
    # アイコン・表示設定
    building_icon = <building_key>        # 背景アイコン
    icon = <job_key>                      # 職業アイコン（省略時は自身）
    condition_string = <loc_key>          # 雇用条件の説明
    
    # 服装インデックス（オプション）
    clothes_texture_index = <int>
    
    # タグ（AI用ヒント）
    tags = { amenities crime energy food minerals alloys unity research }
    
    # 性能チェック用プリトリガー
    possible_pre_triggers = {
        has_owner = <yes/no>
        is_enslaved = <yes/no>
        is_being_purged = <yes/no>
        is_being_assimilated = <yes/no>
        has_planet = <yes/no>
        is_sapient = <yes/no>
        is_robotic = <yes/no>
    }

    # バニラ共通の事前判定
    possible_precalc = can_fill_specialist_job
    
    # 雇用条件（POPスコープ）
    possible = { <conditions> }
    
    # リソース生産・維持費（job出力評価文脈。planet/owner を明示して書く）
    resources = {
        category = <economy_category>
        produces = {
            trigger = { <conditions> }
            <resource> = <float>
        }
        upkeep = {
            trigger = { <conditions> }
            <resource> = <float>
        }
    }

    # 宗主国向けの徴発・再配分出力
    overlord_resources = {
        category = <economy_category>
        produces = { <resource> = <float> }
    }
    
    # 惑星修飾子
    planet_modifier = { <modifiers> }
    triggered_planet_modifier = {
        potential = { <conditions> }
        <modifiers>
        mult = <value>
    }
    
    # 帝国修飾子
    country_modifier = { <modifiers> }
    triggered_country_modifier = {
        potential = { <conditions> }
        <modifiers>
    }
    
    # POP修飾子
    pop_modifier = { <modifiers> }
    triggered_pop_modifier = {
        potential = { <conditions> }
        <modifiers>
    }
    
    # 職業ウェイト（POPスコープ）
    weight = {
        weight = <float>
        mult = value:<script>
        modifier = {
            factor = <float>
            <conditions>
        }
    }
    
    # 昇格・降格（カテゴリ変更時）
    promotion = {
        time = <int>
        effect = { <effects> }
    }
    demotion = {
        time = <int>
        effect = { <effects> }
    }
    
    # 動的データ（外見・名前の変更）
    swappable_data = {
        default = {
            name = <loc_key>
            desc = <loc_key>
            icon = <job_key>
            building_icon = <building_key>
            condition_string = <loc_key>
            weight = <int>
        }
        swap_type = {
            trigger = { <conditions> }
            name = <loc_key>
            desc = <loc_key>
            icon = <job_key>
            building_icon = <building_key>
            weight = <int>
        }
    }
    
    # インラインスクリプト
    inline_script = "<script_path>"
}
```

### 重要プロパティ

#### is_capped_by_modifier

- `yes`（デフォルト）: 建物・区画等の修飾子による職業枠が必要
- `no`: 条件を満たせば常に利用可能。失業が発生しない

#### possible_pre_triggers

パフォーマンス最適化用。全 POP に対して `possible` を評価する前に、簡易条件でフィルタリングします。

```pdx
possible_pre_triggers = {
    has_owner = yes
    is_enslaved = no
    is_being_purged = no
    is_being_assimilated = no
    is_sapient = yes
}
```

#### possible_precalc

バニラ job は `possible_pre_triggers` に加えて `possible_precalc` をよく使います。これは職業階層ごとの共通前提を事前判定するためのショートカットです。

```pdx
possible_precalc = can_fill_specialist_job
possible_precalc = can_fill_worker_job
possible_precalc = can_fill_drone_job
```

- まず `possible_pre_triggers` と `possible_precalc` で粗く絞り、その後に `possible` で個別条件を足す。
- `possible_pre_triggers` には単純な yes/no 判定を置き、長い権利チェックは `possible` や scripted trigger に寄せる。

#### swappable_data

条件に応じて名前・アイコンを動的に変更できます。

```pdx
swappable_data = {
    default = {
        building_icon = building_residence
    }
    swap_type = {
        trigger = {
            has_modifier = resort_colony
        }
        name = tourist
        desc = tourist_desc
        icon = trader
        building_icon = building_amusement_megaplex
        weight = 6
    }
}
```

#### overlord_resources

宗主国契約や徴発で job 出力の一部を上位国へ流す場合は `overlord_resources` を使います。

```pdx
overlord_resources = {
    category = planet_requisitioned_research
    produces = {
        trigger = {
            planet = { has_planet_flag = has_science_ministry }
        }
        physics_research = 0.75
    }
}
```

- 通常の `resources` と別枠で定義する。
- 宗主国側出力の調整も economic category に従う。
- 契約系を触るときは job 本体だけでなく `overlord_resources` の有無も確認する。

### 職業の修飾子

すべての職業は以下の修飾子を自動生成します：

| 修飾子 | 説明 |
|--------|------|
| `job_<key>_add = <int>` | 惑星に職業枠を追加 |
| `job_<key>_per_pop = <float>` | POP数 × 値 の職業枠を追加 |
| `job_<key>_per_crime = <float>` | 犯罪率 × 値 の職業枠を追加 |

### バニラ例: 研究者

```pdx
researcher = {
    category = specialist
    condition_string = SPECIALIST_JOB_TRIGGER
    building_icon = building_research_lab_1
    clothes_texture_index = 3
    
    tags = { research }
    
    possible_pre_triggers = {
        has_owner = yes
        is_being_purged = no
        is_being_assimilated = no
        is_sapient = yes
    }
    
    possible = {
        complex_specialist_job_check_trigger = yes
    }
    
    resources = {
        category = planet_researchers
        produces = {
            physics_research = 4
            engineering_research = 4
            society_research = 4
        }
        upkeep = {
            consumer_goods = 2
        }
    }
    
    weight = {
        weight = @specialist_job_weight
        mult = value:job_weights_research_modifier|JOB|researcher|
        modifier = {
            factor = 0.2
            has_citizenship_rights = no
            NOT = { has_trait = trait_mechanical }
        }
    }
}
```

## 経済カテゴリとの連携

職業の生産量を修飾子で変更するには、`economy_categories` を確認します。

### 生成される修飾子タイプ

- `produces_mult`: 生産量倍率（例: `job_researchers_physics_research_produces_mult`）
- `produces_add`: 生産量加算（例: `job_miners_minerals_produces_add`）

### 例: 採掘師に希少資源を追加

採掘師の経済カテゴリ `planet_miners` が `produces_add` を生成する場合：

```pdx
# 採掘師に Minor Artifacts +1 を追加
modifier = {
    job_miners_minor_artifacts_produces_add = 1
}
```

> [!CAUTION]
> 一部の修飾子（例: `planet_bureaucrats_produces_mult`）は表示されますが、実際には機能しません。

## 実装上の注意

- 複数の `planet_modifier = {}` を同一 job に並べると、最後のものだけが有効になる前提で組む。条件差分は `triggered_planet_modifier` に分解する。
- `triggered_planet_modifier` / `triggered_country_modifier` は `mult` を併用できる。アメニティや交易価値のように job 数連動の値を掛けたいときに使う。
- job の `resources` 内 `trigger` は、`planet = {}`、`owner = {}`、`solar_system = {}` を明示して scope 事故を避ける。
- job 産出の調整を見落としやすい箇所は `resources`、`overlord_resources`、`triggered_*_modifier`、economic category の 4 箇所。

---

## アメニティ・交易価値などの修飾子

職業が生産するアメニティ・交易価値・犯罪削減・海軍許容量などは `triggered_planet_modifier` や `country_modifier` で定義します。これらは職業を上書きしない限り変更できません。

しかし、`_mult` 修飾子を使って帝国・惑星全体の値を乗算することは可能です。

## Q&A

### 専門家が労働者職に就かないのはなぜ？

専門家（rank = 3）は労働者（rank = 2）より高ランクのため、低ランクの職業を受け入れません。

複雑ドローンと単純ドローンは同じ rank = 3/2 でも、集合意識では異なる挙動をします。

### 降格の仕組みは？

1. `keep_from_former_job = yes` の階層では、失業しても元の階層を維持
2. `demotion_time` 日数経過まで、同等以上のランクの職業のみ受け入れ
3. 時間経過後、労働者相当にリセット

### 職業の生産量を変更するには？

- **リソース**: 経済カテゴリの `produces_mult` / `produces_add` 修飾子を使用
- **修飾子（アメニティ等）**: 職業を上書きするか、`_mult` 修飾子で全体を乗算
