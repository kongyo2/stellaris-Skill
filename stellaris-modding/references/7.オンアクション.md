# On Actions (オンアクション)

## 概要

`On Actions`（オンアクション）は、Stellarisのスクリプトシステムにおいて、特定のゲーム内アクションが発生した際に自動的にイベントをトリガーするための仕組みです。これは、イベントを特定の状況にフックさせるための強力な方法であり、イベントドリブンなロジックを実装する上で中心的な役割を果たします。

**例：**
- 「惑星の調査が完了した時 (`on_survey`)」
- 「艦船が破壊された時 (`on_ship_destroyed_victim`)」
- 「宇宙戦闘に勝利した時 (`on_space_battle_won`)」
- 「毎月のパルス (`on_monthly_pulse`)」

## On Actionの構造

オンアクションは `common/on_actions/` ディレクトリ内の `.txt` ファイルで定義されます。各オンアクションは、`events` ブロックと `random_events` ブロックを持つことができます。

```pdx
# /common/on_actions/00_my_on_actions.txt

on_survey = {
    # eventsブロック内のイベントは、トリガーが満たされれば全て実行される可能性がある
    events = {
        my_events.1
        my_events.2
    }

    # random_eventsブロック内のイベントは、重み付けに基づいて一つだけが選ばれて実行される
    random_events = {
        200 = 0             # 200の重みで「何もしない」
        10 = my_random_event.1
        10 = my_random_event.2
    }
}
```

### 実行ロジック

オンアクションが発火すると、以下の処理が実行されます。

1. **`events = { ... }`**:
   - ブロック内にリストされた全てのイベントの `trigger` が評価されます。
   - `trigger` が真 (true) となったイベントが**全て**実行されます。

2. **`random_events = { ... }`**:
   - ブロック内にリストされた全てのイベントの `trigger` が評価され、偽 (false) のものは候補から除外されます。
   - 残った有効なイベントの中から、それぞれの重み付けに基づいて**ランダムに一つだけ**が選ばれ、実行されます。
   - `random_list` とは異なり、`random_events` ブロック自体は `modifier` による重み変更をサポートしていません。しかし、イベントファイル側で `weight_multiplier = { ... }` ブロックを定義することで、動的に重みを変更することは可能です。

### 実行順序

同じオンアクションに複数のファイルからイベントが登録されている場合、それらは**ファイル名のASCII文字コード順**に実行されます。例えば、`aaa_on_actions.txt` に登録されたイベントは `bbb_on_actions.txt` のイベントより先に評価・実行されます。イベントIDの順序は関係ありません。

## パフォーマンスに関するベストプラクティス

特に `planet`, `system`, `starbase`, `leader`, `pop` スコープで頻繁に発火するオンアクションに登録するイベントでは、パフォーマンスへの配慮が重要です。

**`pre_triggers`** を使用することが強く推奨されます。これは、通常の `trigger` よりも先に評価される、より軽量な条件ブロックです。ここで偽と判定されれば、重い `trigger` ブロックの評価をスキップできるため、ゲーム全体のパフォーマンス向上に繋がります。

```pdx
# events/my_planet_event.txt
planet_event = {
    id = my_events.1
    is_triggered_only = yes

    # 先に評価される軽量なトリガー
    pre_triggers = {
        has_owner = yes
        owner = { is_ai = no }
    }

    # pre_triggersを通過した場合のみ評価される、より複雑なトリガー
    trigger = {
        # ... complex conditions ...
    }

    # ... effects ...
}
```

## カスタムOn Action

バージョン3.0以降、スクリプト内で独自に定義したオンアクションを発火させることができます。これは `fire_on_action` エフェクトを使用します。

```pdx
# どこかのエフェクト内でカスタムオンアクションを発火させる
some_effect = {
    fire_on_action = {
        on_action = "my_custom_on_action"
        # スコープを渡すことも可能
        scopes = {
            from = this # thisは fire_on_action を呼び出したスコープ
            fromfrom = root
        }
    }
}
```

そして、このカスタムオンアクションにイベントを登録します。

```pdx
# /common/on_actions/my_custom_on_actions.txt
my_custom_on_action = {
    events = {
        my_custom_event.1
    }
}
```

## 使用例

### 基本的なオンアクション登録

```pdx
# common/on_actions/my_on_actions.txt

# 毎月のパルスにイベントを登録
on_monthly_pulse = {
    events = {
        my_events.monthly_check
    }
}

# 惑星入植完了時にイベントを登録
on_colonized = {
    events = {
        my_events.colony_celebration
    }
    random_events = {
        80 = 0
        20 = my_events.colony_surprise
    }
}
```

### カスタムオンアクションの使用

```pdx
# common/on_actions/my_custom_on_actions.txt
on_my_special_event = {
    events = {
        my_events.special_handler
    }
}

# イベント内でカスタムオンアクションを発火
country_event = {
    id = my_events.trigger_special
    
    immediate = {
        fire_on_action = {
            on_action = on_my_special_event
            scopes = {
                from = root
            }
        }
    }
}
```

### パフォーマンス最適化された例

```pdx
# events/my_events.txt
planet_event = {
    id = my_events.optimized
    is_triggered_only = yes
    
    # 軽量な事前チェック
    pre_triggers = {
        has_owner = yes
        is_occupied = no
    }
    
    # 複雑なトリガーはpre_triggersを通過後のみ評価
    trigger = {
        owner = {
            is_ai = no
            has_technology = tech_example
        }
        any_owned_pop = {
            has_trait = trait_example
        }
    }
    
    # 効果
    immediate = {
        # ...
    }
}
```

## 注意事項

1. **パフォーマンス**: 頻繁に発火するオンアクション（`on_monthly_pulse` など）にイベントを登録する場合は、`pre_triggers` を使用してパフォーマンスを最適化してください。

2. **ファイル順序**: 同じオンアクションに複数のファイルからイベントを登録すると、ファイル名のASCII順に実行されます。実行順序が重要な場合は、ファイル名の先頭に数字を付けることを検討してください（例: `00_`, `01_`）。

3. **スコープの確認**: 各オンアクションには固有のスコープがあります。イベントを登録する前に、`this`, `from`, `fromfrom` などのスコープが何を指すか確認してください。

4. **is_triggered_only**: オンアクションに登録するイベントには、通常 `is_triggered_only = yes` を設定します。これにより、イベントがMTTH（平均発生時間）による発火を防ぎます。

5. **カスタムオンアクション**: スクリプト3.0以降では `fire_on_action` を使用してカスタムオンアクションを発火できます。これにより、モジュラーなイベントシステムを構築できます。
