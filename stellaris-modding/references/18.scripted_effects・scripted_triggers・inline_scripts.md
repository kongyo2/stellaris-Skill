# scripted_effects・scripted_triggers・inline_scripts

Stellaris のバニラは、重複ロジックをそのまま複製せず、`scripted_effects`、`scripted_triggers`、`inline_script` に分解している。共通化の粒度を間違えると scope 事故や追跡不能な差分を生むので、役割を分けて考える。

## 使い分け

| 手段 | 役割 | 向いている用途 |
|---|---|---|
| `scripted_effects` | 再利用可能な処理ブロック | 生成、配布、複数 step の state change |
| `scripted_triggers` | 再利用可能な真偽判定 | rights 判定、職業適性、共有条件 |
| `inline_script` | 小さい断片の差し込み | jobs 追加、output 断片、tooltip 断片、weight 共有 |

## scripted_effects

向いているケース:

- 同じエフェクト列を複数ドメインから呼ぶ。
- パラメータ差し替えで再利用したい。
- scope 遷移を含む重い処理を一箇所で保守したい。

バニラ的な使い方:

- country / fleet / species を生成して event target や flag を保存する。
- 惑星クラスの段階的変更のような if/else 連鎖を共通化する。
- 重いループの先頭に `optimize_memory` を置く。

注意点:

- 呼び出し元 scope を壊さない設計にする。
- パラメータは `$NAME$` 置換で明示する。
- 何を保存し、何を副作用として残すかを分かる形にする。

## scripted_triggers

向いているケース:

- 同じ権利判定や empire type 判定を複数箇所で使う。
- 長い `OR` / `NAND` / `NOR` 条件を一箇所に寄せたい。
- job / species / leader の適格条件を共有したい。

バニラ的な使い方:

- ロボット権利のような制度判定を 1 つの trigger に閉じ込める。
- portrait や species class 系の長い条件列を名前付き判定へ退避する。
- jobs / buildings / events の `possible` や `weight_modifier` から呼ぶ。

注意点:

- 「何を前提 scope とするか」を揃える。
- `from` 依存 trigger は、呼び出し側にも同じ前提を要求する。
- 長い shared trigger ほど名前で意味が伝わるようにする。

## inline_script

向いているケース:

- 建造物や job で繰り返される小断片を差し込みたい。
- 同じ jobs 追加や output ブロックを複数定義で共有したい。
- description / weight / resources の一部を共通化したい。

バニラ的な使い方:

- capital building 群で共通 jobs を差し込む。
- job output の追加部分を研究・職業別に共有する。
- rare technology の `weight_modifier` 断片を共有する。

注意点:

- 小断片向けと割り切る。
- 大きな制御フローまで抱かせると追跡しづらい。
- 呼び出し側の見通しが落ちるなら `scripted_effects` へ上げる。

## どれを選ぶか

- 状態変更を再利用したい: `scripted_effects`
- 条件判定を再利用したい: `scripted_triggers`
- 断片を埋め込みたい: `inline_script`

## 実務チェック

1. その共通化は scope 前提を増やしすぎていないか。
2. パラメータ名だけで意味が分かるか。
3. 個別定義の読みやすさを壊していないか。
4. tooltip 上で見えない効果が増えるなら、別途説明が必要か。
